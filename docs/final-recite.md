# 软件质量与管理 · 课程总结必背

> 围绕课程总结 PDF 的三条主线组织：**过程线 → 项目管理线 → 质量管理线**，可直接背诵。

---

## 一、基础概念

### 1.1 管理三要素

**目标、状态、纠偏。**

- 软件开发天然"熵增"：需求膨胀、沟通混乱、设计退化、缺陷累积。
- 管理通过计划、跟踪、评审、配置管理、度量、过程改进让三要素闭环。

### 1.2 软件危机与软件工程

- **软件危机**：落后的软件生产方式无法满足迅速增长的软件需求。1960 年代，NATO 1968 会议正式提出。
- **软件工程**：研究用工程化方法构建和维护有效的、实用的和高质量的软件的学科。
  - 管理视角：能否复制成功？
  - 技术视角：是否可以将问题解决得更好？
- **广义软件过程** = 技术 + 人员 + 狭义过程。同义词：软件开发方法、软件开发过程。

### 1.3 软件项目管理 vs 软件过程管理

| | 软件项目管理 | 软件过程管理 |
|---|---|---|
| 关注 | 当前项目目标（成本、质量、工期） | 组织/团队的长期过程能力 |
| 类比 | 产品生产管理 | 流水线的设计、维护和升级 |

二者相关但不等同。

### 1.4 Brooks 四大本质困难

**复杂性、不可见性、可变性、一致性。**

- 根植于软件特性，不会被工具或语言进步彻底消除。
- ⚠️ "质量挑战"不是本质困难。

### 1.5 软件发展三大阶段

| 阶段 | 特点 | 典型方法 |
|---|---|---|
| 软硬件一体化 | 软件依附硬件，功能单一，需求变更少 | Measure twice cut once、Code and Fix、线性顺序 |
| 软件成为独立产品 | 操作系统推动独立，功能增强，市场压力 | 形式化方法、结构化程序设计、瀑布模型、成熟度运动 |
| 网络化和服务化 | 规模大，快速演化，需求不确定 | 迭代式开发、敏捷方法、开源开发、DevOps |

## 二、过程线

### 2.1 生命周期模型 vs 软件过程

- **生命周期模型**：对开发过程的人为划分，粗粒度框架，通常不含具体技术实践。
- **软件过程**：包含具体活动、角色、工件、技术实践和管理实践。

### 2.2 瀑布模型

瀑布模型**不是单一模型**，而是一系列模型。项目挑战越多，所需过程元素越复杂。常见失败原因：团队低估挑战，选择了过于简单的瀑布模型。

### 2.3 迭代式开发

把软件开发分成多个小周期，每轮都经历需求、设计、实现、验证。目标：尽早获得反馈、逐步降低不确定性。迭代式开发是**方法论族**，不是上世纪 90 年代才出现。

### 2.4 PDCA 与 IDEAL

**PDCA（戴明环）**：

| P（计划） | D（执行） | C（检查） | A（处理） |
|---|---|---|---|
| 分析现状→分析原因→找主因→定措施 | 执行措施和计划 | 检查执行效果 | 成功经验标准化；未解决问题进入下一轮 |

**IDEAL**：Initiating → Diagnosing → Establishing → Acting → Leveraging

⚠️ PDCA/IDEAL 是过程改进参考模型，**可以用于敏捷环境**。

### 2.5 CMMI 五级

| 等级 | 名称 | 核心特征 |
|---|---|---|
| 1 | Initial 原始级 | 混乱，依赖个人英雄主义，救火文化 |
| 2 | Managed 已管理级 | 项目层面有计划、跟踪、需求管理、配置管理 |
| 3 | Defined 已定义级 | 组织层面有标准过程，项目可裁剪 |
| 4 | Quantitatively Managed 定量管理级 | 统计过程控制(SPC)和预测模型 |
| 5 | Optimizing 优化级 | 持续识别偏差、根因并改进 |



**为什么 4/5 级是高等级**：
- 2/3 级关注**当前状态**——过程是否按计划执行。
- 4 级用统计过程控制和预测模型，能根据量化数据**预判未来**。
- 5 级主动识别偏差、找到根因并消除，实现**持续优化**。

**CMMI 与敏捷不对立**：CMMI 是过程管理/改进模型，XP/Scrum 是开发方法或过程框架，二者性质不同。

### 2.6 开源软件开发

- **Linus 定律**："如果有足够多的 beta 测试者和合作开发者，几乎所有问题都会很快显现，然后自然有人会把它解决。"
- 核心实践："早发布，常发布，倾听用户反馈"；"把用户当成开发合作者"。
- 演化形式：内部开源（Inner Source）、众包（Crowdsourcing）。

## 三、敏捷宣言与核心概念

### 3.1 敏捷宣言四条

1. **个体和互动** 胜过 流程和工具
2. **可以工作的软件** 胜过 详尽的文档
3. **客户合作** 胜过 合同谈判
4. **响应变化** 胜过 遵循计划

> 尽管右项有其价值，但更重视左项的价值。

### 3.2 敏捷核心概念

- **诞生**：2001 年，犹他州雪鸟滑雪场，17 位方法论专家签署。
- **价值观-原则-实践**体系：价值观是最根本层次。
- 敏捷方法是**适应性而非预见性**的、**面向人而非面向过程**的。
- 工程方法（计划驱动）借鉴**土木工程**思路，将"设计与建造分离"。

### 3.3 关键人物与数据

- **Alistair Cockburn**：软件开发中的"人"是"**非线性、一阶的部件**"——不能像制造业把人当可替换插件。
- **McConnell**：大型软件项目中编码和单元测试仅占约 **15%**。
- **度量失效（古德哈特定律）**：度量若没涵盖所有重要因素，人会改变工作方式迎合指标，造成度量失效。

### 3.4 Standish Group 项目分类

- **Successful**：按时按预算完成所有功能。
- **Challenged**：已完成可运行，但超预算/超期/功能少于规格。
- **Impaired/Failed**：开发中途被取消。
- 敏捷观点：**为客户创造价值**才是项目成功的最重要标准。

### 3.5 不要误解敏捷

- "轻量级""不计划""不严格"都容易误导。
- 优秀敏捷方法也很严格；所有正式项目都需要计划。
- 计划驱动与敏捷**不是绝对反义词**。

## 四、Scrum

### 4.1 核心记忆：33355

- **3 角色**：Product Owner、Scrum Master、Developers
- **3 工件**：Product Backlog、Sprint Backlog、Increment
- **3 支柱**：透明、检视、适应
- **5 事件**：Sprint、Sprint Planning、Daily Scrum、Sprint Review、Sprint Retrospective
- **5 价值观**：承诺、专注、开放、尊重、勇气



### 4.2 关键数字

- 术语来源：竹内弘高与野中郁次郎，**1986** 年。
- Scrum Team 通常 **10 人或更少**。
- Daily Scrum 时间盒：**15 分钟**。
- Sprint 周期：约 **2 周**，最长不超过一个月。
- 理论基础：**经验主义与精益思维**。

### 4.3 角色职责

- **Product Owner**：对产品价值最大化负责；管理 Product Backlog（内容、排序、透明度）；**排序权归 PO**。
- **Scrum Master**：保证 Scrum 被正确理解和使用；服务团队、PO 和组织；移除障碍。
- **Developers**：每个 Sprint 创建可用 Increment；自管理完成工作；保证质量符合 DoD。

### 4.4 用户故事

- 模板：`作为<某类用户>，我想<做某事>，从而<获得某种价值>` → who / what / why
- **3C**：Card（卡片）、Conversation（交谈）、Confirmation（确认）
- **INVEST**：**I**ndependent、**N**egotiable、**V**aluable、**E**stimable、**S**mall、**T**estable

### 4.5 猪与鸡、DoD

- **猪**（全身投入）：PO、SM、Developers
- **鸡**（参与但不决策）：用户、客户、经理等
- **DoD**：团队对"完成"的统一定义，判断增量是否真正可交付。

### 4.6 大规模敏捷

- **SAFe**：团队层 → 项目群层(ART) → 解决方案层 → 组合层。争议："披着敏捷外衣的瀑布"。
- **LeSS**：基本 LeSS（2-8 团队）、LeSS Huge（8+ 团队）。一个 PO、共享 Backlog、同一 Sprint。

## 五、XP（极限编程）

### 5.1 核心记忆

- 起源项目：**克莱斯勒 C3 薪资系统**（Smalltalk，8.7 万员工）。
- 五项价值观：沟通、简单、反馈、勇气、尊重（第 2 版新增）。
- 四项基本活动：**编码、测试、倾听、设计**。

### 5.2 关键实践

- **TDD**：Red → Green → Refactor（红 → 绿 → 重构）
- **简单设计四规则**（按优先级）：①通过所有测试 → ②消除重复 → ③清晰表达意图 → ④最小化元素
- **重构**：保留外部可见行为、只改善内部结构。改变行为是功能变更。
- **YAGNI**：存疑时默认不上。简单设计以重构能力 + 自动化测试为底气。
- **结对编程**：Driver 关注当下实现，Navigator 关注整体方向。
- **40 小时工作制**：反对长期加班掩盖计划/范围/设计/测试问题。

### 5.3 变更成本曲线

- Boehm 1981：需求 1x → 设计 5x → 编码 10x → 测试 20x → 上线 100x
- Kent Beck 押注"**把指数线拉平**"——通过 TDD + CI + 重构把反馈周期从几个月缩到几分钟。
- **平坦曲线五根支柱**：面向对象、简单设计、自动化测试、重构技术、CI/CD。拿掉任一支柱，曲线立刻陡回去。

### 5.4 结对编程经济账

成本：**+15%** 开发时间。收益：**-15%** 缺陷；知识实时传递；复杂任务设计质量更高。

### 5.5 TDD 实证数据

- 缺陷密度：比非 TDD 低约 **45%**。
- 初始开发时间：比非 TDD 高约 **16-30%**。
- 3 年维护成本：比非 TDD 低约 **50%+**。

### 5.6 Dark Scrum 警示

- Ron Jeffries：只有 Scrum 仪式没有 XP 工程实践的团队。
- Fowler 2018："**没有技术卓越的敏捷，是一种赶死加速。**"

## 六、Kanban 与精益

### 6.1 Kanban 六大实践

1. 可视化工作流
2. **限制 WIP**（核心）
3. 管理流动
4. 显式化流程规则
5. 建立反馈循环
6. 协作式改进（实验性演进）



### 6.2 核心度量

- **WIP**：已开始但未完成的工作项数量。
- **Throughput（产能）**：每单位时间完成的工作项数量。
- **Cycle Time（周期时间）**：工作项从正式开始到完成的时长。
- **Lead Time（交付时间）**：含需求提出到启动前的等待时间，**Lead Time ≥ Cycle Time**。
- **SLE**：服务水平期望（如 "85% 工作项在 8 天内完成"）。
- **WIP 上限**：团队人数 × 2/3 ~ 3/4。

### 6.3 Little 定律

```
L = λW
```

- L：平均在制品数量
- λ：平均到达率
- W：平均周期时间



### 6.4 Kanban vs Scrum

| | Scrum | Kanban |
|---|---|---|
| 角色 | 固定（PO/SM/Dev） | 不强制 |
| 迭代 | 固定 Sprint 时间盒 | 不强制迭代 |
| 变更 | Sprint 内通常不接受 | 可连续拉取和调整 |
| 适用 | 产品开发 | 持续流、维护、Ops |



### 6.5 精益思想

**精益屋两大支柱**：
- **JIT（准时化）**：拉动式系统，按下游需求触发上游生产。
- **Jidoka（自働化）**：⚠️ "働"比"動"多"亻"旁，强调**带有人类判断的自动化**（autonomation ≠ automation）。

**精益演化**：福特流水线(1913) → 丰田 TPS(1940s-50s) → Poppendieck 精益软件开发(2003)

**精益七大原则（Poppendieck）**：
1. 消除浪费
2. 内建质量
3. 创建知识
4. **推迟决策**（最后责任时刻 LRM）
5. 快速交付（最有效手段：严格限制 WIP）
6. 尊重人（自主、专精、目的）
7. 全局优化（局部最优 ≠ 全局最优）

**七种软件浪费**：部分完成的工作、额外功能、重新学习、交接、延迟、任务切换、缺陷

## 七、估算、计划与跟踪

### 7.1 估算要点

- 估算本质上是猜测，目标是提高**一致性和可信度**。
- 规模估算更适合利用历史数据（偏差原因相对客观）。
- 时间估算受主观能动性影响，历史时间数据不一定可靠。
- **估算要的是过程，而非结果**——估算过程是干系人达成共识的过程。
- 目的：支撑管理决策、理解范围复杂度、提供资源/工期依据、建立跟踪基线。

### 7.2 PROBE 方法

**PROBE** = Proxy Based Estimation（基于代理的估算）

**基本思想**：用代理规模连接早期规划需要和后期精确度量；估算相对大小；依赖高质量历史数据和一致过程。

**流程**：概要设计 → 代理识别和代理规模(E) → 估算并调整程序规模 → 计算预测区间；同时 → 估算并调整资源 → 计算预测区间。

**为什么不用历史生产效率数据**：
- 生产效率在分母上，个体软件工程师生产效率波动大。
- 直接用历史生产率倒推会放大估算偏差。
- 所以用代理规模→规模的回归代替。

**优点**：过程清楚，可积累数据，可用统计方法调整。
**缺点**：高度依赖历史数据质量，数据缺失或不稳定时偏差显著。

### 7.3 通用计划框架

链路：客户需求 → 定义需求 → 概要设计 → 规模估算 → 资源估算 → 日程计划 → 开发产品 → 收集数据 → 过程分析

- **正推**，不是从截止日期倒推。
- 需求定义和概要设计需人工判断。
- 规模和资源估算借助历史数据和模型。

### 7.4 风险应对策略

| 策略 | 含义 |
|---|---|
| 规避 | 改变计划以消除风险 |
| 转移 | 将风险影响和责任转移给第三方 |
| 缓解 | 降低风险发生概率或减少影响 |
| 接受 | 不主动改变计划，准备发生时处理 |



### 7.5 挣值管理（EVM）

**核心变量**：PV（计划值）、EV（挣值）、AC（实际成本）、BAC（完工预算）

**常用公式**：
```
SV = EV - PV          （进度偏差，>0 超前）
SPI = EV / PV         （进度绩效指数，>1 超前）
CV = EV - AC          （成本偏差，>0 节约）
CPI = EV / AC         （成本绩效指数，>1 节约）
EAC = AC + (BAC - EV)/CPI
EAC = BAC / CPI
```

**三种实现**：

| 层次 | 信息 | 能回答的问题 |
|---|---|---|
| 简单 | WBS + 计划价值 + 完成状态 | 完成了多少有价值的工作？ |
| 中级 | + PV | 进度超前还是落后？⚠️ 不引入成本 |
| 高级 | + AC + BAC | 成本节约还是超支？预计完工成本？ |

⚠️ 中级实现不引入成本信息，成本信息属于高级实现。EV 强调**完成任务才获得价值**，比只看工时更能反映真实进展。

## 八、团队与人员管理

### 8.1 知识工作与自主团队

软件开发是复杂知识工作，开发者是智力劳动者。

**自主团队六大特征**：
1. 自行定义项目目标
2. 自行决定团队组成和角色
3. 自行决定开发策略
4. 自行定义开发过程
5. 自行制定开发计划
6. 自行度量、管理和控制项目工作

### 8.2 Maslow 需求层次（五层）

1. 生理需求 → 2. 安全需求 → 3. 社交/归属需求 → 4. 尊重需求（Esteem） → 5. 自我实现（Self-actualization）

⚠️ 自我实现 ≠ 自尊（第 5 层 ≠ 第 4 层）。晚年修正：高层次需求不一定要等低层次完全满足后才出现。

### 8.3 期望理论

`Motivation = Valence × Expectancy`

- 过于宏大的目标提高价值感但降低可达成预期，反而削弱动力。
- "构建高质量软件"太抽象，应拆成可执行、可跟踪的里程碑。

### 8.4 其他激励理论

- **McGregor X/Y**：X 理论认为人不喜欢工作需管控；Y 理论认为人能自我导向追求责任。
- **Herzberg 双因素**：激励因素（成就感、认可）vs 保健因素（薪金、环境）。
- **三种领导方式**：威逼 → 利诱（交易型）→ **鼓励承诺（转变型）**——知识工作首选。
- **承诺四条件**：自愿、公开、可信、向团队承诺。

### 8.5 TSP 角色与启动

**六大角色**：
- **项目组长**：建设和维持高效团队，激励成员，主持会议，汇报状态
- **计划经理**：开发团队和个人计划，平衡计划，跟踪进度
- **开发经理**：制定开发策略，规模和资源估算，负责开发活动
- **质量经理**：制定和跟踪质量计划，警示质量问题，组织评审
- **过程经理**：维护过程数据、开发标准、会议记录，支持过程改进
- **支持经理**：保证工具环境，配置管理，维护风险/问题跟踪

**TSP Launch 10 次会议**：
1. 建立产品和业务目标 → 2. 分配团队角色 → 3. **确定开发策略**（常考）→ 4. 制定总体计划 → 5. 制定质量计划 → 6. 制定个人计划 → 7. 风险评估 → 8. 计划再平衡 → 9. 管理层评审 → 10. 项目启动

## 九、质量管理

### 9.1 质量管理三要素挑战

- **目标**：质量目标不容易被单一指标完整表达，不同用户优先级不同。
- **状态**：质量状态不完全可见（缺陷、可维护性隐藏在代码内部）。
- **纠偏**：越晚纠偏成本越高，需把评审、测试、度量前移。

### 9.2 面向用户的质量观

质量 = **满足用户需求的程度**，最终从用户价值和满意度判断。

**答题四问**：
1. 用户是谁？
2. 用户需求的优先级是什么？
3. 这种优先级会怎样影响开发过程？
4. 怎样度量这种质量观下的质量水平？

### 9.3 质量管理策略

核心思想：用**缺陷管理替代抽象的质量管理**；尽早通过高质量评审消除缺陷。

- 缺陷存在越久，消除成本越高。
- 代码评审/设计评审通常比后期测试更高效。
- 测试不能被取消，但应尽量让缺陷在上游阶段被发现。
- 测试发现缺陷多 = 前期质量差，不应把"测出很多 bug"当作质量好。

### 9.4 PSP 基本度量元

三个基本度量元：**规模（Size）、时间（Time）、缺陷（Defect）**。

⚠️ "风险"不是 PSP 基本度量元。

### 9.5 GQM

**G**oal → **Q**uestion → **M**etric。先明确目标，再提出问题，最后定义度量元。

### 9.6 A/FR（质检失效比）

```
A/FR = PSP 质检成本 / PSP 失效成本
     = (设计评审时间 + 代码评审时间) / (编译时间 + 单元测试时间)
```

- PSP 期望值约为 **2.0**。
- 过低：评审不足。过高：可能评审过度、效率下降。

### 9.7 PQI（过程质量指标）

五个因子（每个 0~1，越接近 1 越好）：

1. **设计质量**：设计时间应 > 编码时间
2. **设计评审质量**：设计评审时间应 > 设计时间的 50%
3. **代码评审质量**：代码评审时间应 > 编码时间的 50%
4. **代码质量**：编译缺陷密度应低
5. **程序质量**：单元测试缺陷密度应低

```
PQI = 设计质量 × 设计评审质量 × 代码评审质量 × 代码质量 × 程序质量
```

⚠️ PQI 越接近 1 越好，五个分指标不能超过 1.0。

### 9.8 Yield（缺陷消除率）

- **Phase Yield**：某一阶段消除缺陷的比例。
- **Process Yield**：过程整体消除缺陷的比例。

Yield 可用于缺陷预测模型和过程改进。

### 9.9 DRL（缺陷消除效率比）

```
DRL = 某阶段每小时发现缺陷数 / 单元测试每小时发现缺陷数
```

⚠️ 以**单元测试每小时发现缺陷率**作为基准（不是缺陷个数）。

### 9.10 Review Rate（评审速度）

- 代码评审：不超过 **200 LOC/小时**。
- 文档评审：不超过 **4 页/小时**。

### 9.11 Quality Journey（质量路径）

**顺序**：稳定测试 → 提升进入测试前的产物质量 → 度量并稳定团队评审 → 建立质量意识 → 度量并稳定个人评审 → 诉诸设计 → 设计验证 → 缺陷预防 → 走向用户质量观

- 哲学：先把可见的质量活动稳定，再逐步前移到评审、设计和预防。
- 从被动发现缺陷 → 主动预防缺陷。

### 9.12 个人评审与小组评审

- **个人评审**：工程师在编译和单元测试前，依据检查表独立阅读**自己的代码**。
- **时机**：在编译、单元测试**之前**发现缺陷。
- **小组评审**：适合发现理解偏差、接口不一致、设计缺陷。
- 个人评审不能被小组评审替代；检查表应**个性化**、**定期更新**。

### 9.13 PSP 四大设计模板

| 模板 | 全称 | 描述对象 | 记忆点 |
|---|---|---|---|
| OST | Operational Specification Template | 操作场景、外部交互、正常/异常使用 | **外部行为** |
| FST | Functional Specification Template | 对外接口、函数/方法规格 | **接口静态信息** |
| SST | State Specification Template | 状态、状态转移、转移动作 | **内部动态信息** |
| LST | Logical Specification Template | 内部逻辑、伪代码、形式化逻辑 | **内部静态/逻辑信息** |



### 9.14 设计层次

| 层次 | 关注点 | 对应模板 |
|---|---|---|
| 系统层 | 整体架构、子系统划分、外部接口 | OST |
| 组件层 | 模块/组件间的接口和交互 | FST |
| 模块层 | 单个模块的内部逻辑和状态 | SST（动态）+ LST（静态/逻辑） |



### 9.15 设计验证

**状态机正确性条件**：
- 没有死循环和陷阱
- 状态转换条件满足**完整性**
- 状态转换条件满足**正交性**
- 还要符合设计意图（形式正确 ≠ 业务正确）

**验证手段**：执行表、跟踪表、符号化执行、程序正确性证明

### 9.16 质量管理大师

| 人物 | 核心贡献 |
|---|---|
| **Deming** | PDCA 循环（戴明环），日本质量管理之父。质量应由过程构建出来。 |
| **Juran** | 质量三部曲（计划、控制、改进），适用性质量，80/20 原则。 |
| **Crosby** | 零缺陷思想，质量 = 符合要求，质量来自预防。 |
| **Humphrey** | 软件过程之父，CMM → PSP → TSP，将统计质量控制引入软件工程。 |

## 十、DevOps

### 10.1 DevOps Three Ways

**First Way：从左到右的流动**
- 打通开发→测试→部署→运维→生产的价值流。
- 只有进入生产并被用户使用的功能才真正产生价值。
- 小批量、限制 WIP、持续构建/集成/交付。

**Second Way：从右到左的反馈**
- 生产、运维、测试中的问题快速回流到开发。
- 自动化测试、监控、日志、告警、停止生产线。

**Third Way：持续学习与实验文化**
- 允许安全失败，从失败中学习。
- 鼓励小实验、持续改进、高信任文化。

### 10.2 CI/CD/持续部署 三层递进

| 层级 | 含义 | 关键判断 |
|---|---|---|
| **CI**（持续集成） | 自动构建 + 快速测试，频繁合入主线 | 主线始终"已集成已初步验证" |
| **CD**（持续交付） | 流水线自动到类生产，只剩上线按钮手动 | 主线始终"可随时发布" |
| **持续部署** | 连上线也自动化，只保留紧急刹车 | 发布变成无感事件 |

⚠️ **递进关系**：CI 是 CD 的前提，CD 是持续部署的前提。三者**不是同义词**。

### 10.3 构建流水线分段

| 阶段 | 内容 | 时间 | 规则 |
|---|---|---|---|
| 提交阶段 | 编译 + 单元测试 + 静态检查 | ≤ **10 分钟** | 红灯 → stop the line |
| 集成测试 | 跨模块 + 真实数据库 | ~30 分钟 | 失败快速修不阻塞主线 |
| 端到端测试 | UI + API + 真实场景 | ~60-90 分钟 | 同上 |
| 性能/安全 | 压测 + 安全扫描 | 数小时 | 同上 |



### 10.4 CI 关键判断

- CI **不是工具**，是团队协议：每天至少一次合入共享主线。
- 提交阶段超过 10 分钟 → 开发者心流被打断。
- 主线构建坏了 → 首选 `git revert`，把故障半径压到最小。
- 长期 feature branch 上的自动化构建 **≠ CI**。
- CI 判定标准（Fowler）：**看每天有没有真把代码合回主线**，不看用没用 Jenkins。

### 10.5 未完成功能的处理技术

三者共同点：**把"代码部署"和"功能发布"解耦**

- **Feature Toggle**：代码里加配置开关，未完成功能默认关。
- **Branch by Abstraction**：引入抽象层，新旧实现并存，逐步迁移。
- **Dark Launch**：新功能部署到生产但对用户不可见，内部监控压测。

### 10.6 DevOps 流行原因

- 业务节奏加快，需要更短交付周期。
- 云计算、容器、自动化工具链成熟。
- 敏捷/精益/看板思想向开发后段延伸。
- 微服务需要自动化治理。
- CI/CD 让代码变更更快进入验证和生产。

### 10.7 微服务

- 围绕业务能力拆分，**高内聚、低耦合**。
- 去中心化治理和数据管理。
- 独立部署和发布。
- 微服务本质上是 **SOA 的一种扩展/演进**。

### 10.8 容器与云原生

**云原生四大特征**：快速创新、高可用、弹性扩展、移动优先

**XaaS**：

| 类型 | 供应商管理范围 |
|---|---|
| IaaS | 网络、存储、服务器、虚拟化 |
| PaaS | + 运行环境、中间件、操作系统 |
| SaaS | 全部（含应用和数据） |
| FaaS | Serverless，按函数粒度 |

### 10.9 AI 时代冲击（2026 新增）

- 角色冲击：开发者从代码编写者变为 Agent 编排者。
- **约束理论(TOC)**：优化非瓶颈不会提高产出。
- Faros AI 数据：PR 数量 +98%、PR 规模 +154%、bug 率 +9%。
- **理解力负债**：AI 代码生成速度是人类理解速度的 5-7 倍。
- DoD 可追加："至少一名成员能解释核心设计决策"。

### 10.10 DORA 2024

- 部署频率：Elite vs Low 差距 **182 倍**。
- 变更失败率：Elite vs Low 低 **8 倍**。
- Elite 团队：前置时间 < 1 小时，恢复时间 < 1 小时。
- 50.2% 受访者在 CI/CD 中使用 AI 工具。
