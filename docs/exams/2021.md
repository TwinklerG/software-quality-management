## 一、选择题

### 1. 基于Yield指标构建缺陷预测模型，并列举该模型的可能改进方案


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>总体思想</strong>：利用回归技术预测软件开发过程中各阶段的Inject Rate（缺陷注入率）和Yield（缺陷消除率）。</p>
<p>Yield指标只能用来估算，不可以用来度量。构建模型需要知道：</p>
<ul>
<li>注入阶段注入多少缺陷</li>
<li>缺陷注入的密度（需求每一页注入多少缺陷）</li>
<li>缺陷注入的速度（每小时注入多少缺陷）</li>
<li>消除阶段的缺陷注入密度和速度</li>
<li>历史数据中的软件规模、文档规模、开发人员规模</li>
</ul>
<p><strong>步骤</strong>：</p>
<ol>
<li>确定纳入影响因子的数据以及数据度量方法</li>
<li>从系统历史库中收集历史数据，并进行整理</li>
<li>依照回归技术进行计算</li>
<li>在项目进行过程中不断收集数据，与预测数据进行比较，调整回归参数</li>
<li>项目过程中依据实际数据与预测数据的误差进行风险的预防、识别和控制</li>
</ol>
<p><strong>改进方案</strong>：</p>
<ul>
<li>维护历史数据，保证数据质量（简单性、可理解性、稳定性、可度量性、相关性）</li>
<li>影响因子选择上不仅需要软件规模数据，还需要开发过程、文档、人员等方面的数据</li>
<li>反馈模型：随着实际数据产生，将其作为输入变量调整回归参数</li>
<li><strong>重要改进</strong>：假设注入水平和消除水平都符合正态分布，计算均值和标准差，用蒙特卡罗方法模拟结果</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第五讲.pdf</code>——Yield指标与缺陷预测模型。</p>
</blockquote>

</details>

## 二、简答题

### 1. 我们如何理解瀑布模型（6分）


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<ol>
<li>瀑布模型不是单一模型，是一系列模型，覆盖最简单场景（过程元素少）到最复杂的场景（过程元素多）</li>
<li>软件项目应该结合实际情况选择合适过程元素的瀑布模型，基本原则是，项目面临困难和挑战越多，选择的模型应该越复杂</li>
<li>软件项目团队往往低估项目的挑战，选择了过于简单的不适用的瀑布模型（这是瀑布模型实践中失败的最常见原因）</li>
</ol>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第二讲.pdf</code>——瀑布模型的理解。常见失败原因是低估项目挑战，选择过于简单的瀑布模型，而非"瀑布模型天然无效"。</p>
</blockquote>

</details>

### 2. 给一个挣值管理的表格，项目进度提前还是落后？有可能出现什么风险？


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p>分析挣值管理表格的核心指标：</p>
<ul>
<li><strong>SV（Schedule Variance）= EV − PV</strong>：正值表示进度超前，负值表示进度落后</li>
<li><strong>SPI（Schedule Performance Index）= EV / PV</strong>：&gt;1表示进度超前，&lt;1表示进度落后</li>
<li><strong>CV（Cost Variance）= EV − AC</strong>：正值表示成本节约，负值表示成本超支</li>
<li><strong>CPI（Cost Performance Index）= EV / AC</strong>：&gt;1表示成本节约，&lt;1表示成本超支</li>
</ul>
<p><strong>课上讲的具体例子分析</strong>：</p>
<ul>
<li>通过To-date看，Earned Value to date中Actual &gt; Plan，说明迄今为止进度没有落后</li>
<li>但是，hours to date总共花的时间大于预期，每个小时预期创造0.3个价值，实际上每小时创造0.28个价值，说明实际工作效率不如预期</li>
<li><strong>结论</strong>：在下周或过几周，项目进度可能会逐渐落后，项目开发有隐患</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第四讲.pdf</code>——EVM数据分析和风险识别。</p>
</blockquote>

</details>

## 三、判断题

### 过程改进模型可不可以作为评价公司能力的标准？比如某公司宣称自己达到了CMMI 5级，开发能力比谷歌强，你认为合理吗？


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答：不合理。</strong></p>
<p><strong>理由</strong>：</p>
<ol>
<li>CMMI不是过程优劣的标准，也不适合用作公司之间的能力比较</li>
<li>它是每个公司自我反思改进的依据</li>
<li>由于企业所处环境以及业务目标等的差异，过程改进模型对不同企业的含义不同</li>
<li>例如：银行对数据一致性、可靠性有很高要求，开发模型侧重测试、原型等；而嵌入式公司更关注性能</li>
<li>成熟度等级不应该脱离企业环境直接横向比较；处于相同的成熟度等级，也不能说明这些企业的研发能力是相同的</li>
</ol>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第七讲.pdf</code>——CMMI的应用场景和局限性。</p>
</blockquote>

</details>
