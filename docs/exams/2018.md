## 一、简答题

### 1. 软件发展三大阶段的特点和主流开发方法


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>软硬件一体化</strong>：</p>
<ul>
<li>特点：软件支持硬件完成计算任务、功能单一、复杂度有限、几乎不需要需求变更</li>
<li>主流开发方法：三思而后行（Measure twice, cut once）；Code and Fix（编码+改错）</li>
</ul>
<p><strong>软件成为独立的产品</strong>：</p>
<ul>
<li>特点：摆脱了硬件的束缚（操作系统）、功能强大、个人电脑出现、需求多变、兼容性要求、来自市场的压力</li>
<li>主流开发方法：形式化方法、结构化程序设计和瀑布模型</li>
</ul>
<p><strong>网络化和服务化</strong>：</p>
<ul>
<li>特点：功能更复杂、规模更大、用户数量急剧增加、快速演化和需求不确定、分发方式的变化、进一步的服务化和网络化、盛行开源和共享文化</li>
<li>主流开发方法：迭代式开发、敏捷开发（XP、Scrum、Kanban）、开源软件开发方式、DevOps</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第二讲.pdf</code>——软件发展三大阶段。</p>
</blockquote>

</details>

### 2. 软件项目管理和软件过程管理


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<ul>
<li><strong>软件项目管理</strong>：是应用方法、工具、技术以及人员能力来完成软件项目，实现项目目标的过程。关注当前项目的成本、质量和工期。</li>
<li><strong>软件过程管理</strong>：是为了让软件过程在开发效率、质量等方面有着更好性能绩效。关注组织或团队的软件过程能力，使过程在效率、质量、可预测性等方面持续改进。</li>
</ul>
<p><strong>类比</strong>：项目管理像产品生产管理，过程管理像流水线的设计、维护和升级。二者相关但不等同。</p>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第二讲.pdf</code>——软件项目管理与过程管理。</p>
</blockquote>

</details>

### 3. 生命周期模型与软件过程的区别和联系


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<ol>
<li>生命周期模型是对一个软件开发过程的人为划分</li>
<li>生命周期模型是软件开发过程的框架，是对软件开发过程的一种粗粒度划分</li>
<li>生命周期模型往往不包括技术实践</li>
</ol>
<p><strong>联系</strong>：生命周期模型为软件过程提供框架；软件过程在周期模型内填充具体实践。</p>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第二讲.pdf</code>。</p>
</blockquote>

### 4. 请描述一下PROBE方法的基本原理和过程，并解释在应用PROBE A方法估算时间的时候，为什么不用历史数据中的生产效率数据


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>基本原理</strong>：</p>
<ul>
<li>设立合理的代理作为精确度量和早期规划需要的度量之间的桥梁</li>
<li>关注相对大小，而非绝对大小</li>
<li>依赖高质量的历史数据和一致过程</li>
</ul>
<p><strong>过程</strong>：</p>
<div class="codehilite"><pre><span></span><code>概要设计 → 代理识别和代理规模(E) → 估算并调整程序规模 → 计算预测区间
                                  → 估算并调整资源 → 计算预测区间
</code></pre></div>
<p><strong>为什么不使用生产效率数据</strong>：在估算资源需求（例如人时）的时候，生产效率一般在分母上。考虑到个体软件工程师生产效率波动较大，容易导致估算偏差范围变大。时间估算偏差的原因可能正是估算结果本身（自我实现的预言），使得历史数据中时间偏差的参考价值不大。</p>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第四讲.pdf</code>——PROBE方法。</p>
</blockquote>

</details>

### 5. 谈谈你对项目估算的认识，并简要解释应用PROBE方法估算的优缺点


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>对项目估算的认识</strong>：</p>
<ul>
<li>规模估算往往可以依据历史数据来完成，偏差产生原因相对客观，偏差可以用来修正新的估算结果</li>
<li>时间估算的偏差产生原因更加复杂，一方面和规模有关，另外一方面跟人的主观能动性有关</li>
<li>估算本质上是一种猜测，追求的目标应该是一致性以及估算结果的使用者对估算结果的信心</li>
</ul>
<p><strong>PROBE方法的优点</strong>：</p>
<ul>
<li>通过定义的估算过程和数据收集以及使用的框架，使得估算结果可以尽可能一致</li>
<li>使得一些统计方法可以用来调整估算结果，增强用户对估算结果的信心</li>
</ul>
<p><strong>PROBE方法的缺点</strong>：</p>
<ul>
<li>非常依赖高质量的历史数据</li>
<li>一旦数据不完整或者缺失，就可能导致估算结果有显著偏差</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第四讲.pdf</code>——PROBE估算方法。</p>
</blockquote>

</details>

### 6. 请解释A/FR，PQI的计算方式，并且解释这两个指标有什么用途


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>A/FR（Appraisal to Failure Ratio，质检失效比）</strong>：</p>
<ul>
<li>A/FR = PSP质检成本 / PSP失效成本</li>
<li>质检成本 = 设计评审时间 + 代码评审时间</li>
<li>失效成本 = 编译时间 + 单元测试时间</li>
<li>用途：理论上A/FR值越大，质量越高；但A/FR值过大说明评审过多，开发效率低下。PSP中A/FR期望值为<strong>2.0</strong></li>
</ul>
<p><strong>PQI（Process Quality Index，过程质量指标）</strong>：</p>
<ul>
<li>PQI为5个数据乘积（以基准值作为1，最后结果越接近1，质量越高）：</li>
<li><strong>设计质量</strong>：设计时间应该大于编码时间 — min(设计时间/编码时间, 1)</li>
<li><strong>设计评审质量</strong>：设计评审时间应大于设计时间的50% — min(2×设计评审时间/设计时间, 1)</li>
<li><strong>代码评审质量</strong>：代码评审时间应大于编码时间的50% — min(2×代码评审时间/编码时间, 1)</li>
<li><strong>代码质量</strong>：编译缺陷密度应当小于10个/千行 — min(编译缺陷密度+10)/20, 1)</li>
<li><strong>程序质量</strong>：单元测试缺陷密度应当小于5个/千行 — min(单元测试缺陷密度+5)/10, 1)</li>
</ul>
<p><strong>PQI的用途</strong>：</p>
<ul>
<li>判断模块开发质量</li>
<li>规划质量活动计划</li>
<li>过程改进</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第五讲.pdf</code>——A/FR和PQI指标。</p>
</blockquote>

</details>

### 7. DevOps的特点，为什么流行？


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p>DevOps以敏捷项目管理技术和微服务支持为中心，通过基于版本控制标准的自动化来处理整个软件开发生命周期。</p>
<p><strong>DevOps的好处（流行原因）</strong>：</p>
<ul>
<li><strong>敏捷团队项目管理</strong>：增强网站和移动应用软件开发的管理</li>
<li><strong>软件开发过程的优化</strong>：通过CI/CD功能实现，快速推出新的软件功能</li>
<li><strong>促进协作</strong>：Git允许开发人员在具有订单项回滚能力的团队中进行协作</li>
<li><strong>通过自动化提高效率</strong>：CI/CD通过企业编程工具支持自动化代码测试</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第七讲.pdf</code>——DevOps。</p>
</blockquote>

</details>

## 二、简述题

### 1. 请结合软件开发特点介绍软件项目管理中自主型团队的必要性


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p>软件开发是一种智力活动，开发者是智力劳动者，而对于智力劳动者而言，管理的第一准则就是智力劳动者不能被管理，只能实现自我管理：</p>
<ul>
<li>处理和讨论非常抽象的概念</li>
<li>把不同的部分整合成一个可以工作的正确的系统</li>
<li>全身心地参与</li>
<li>努力做出卓越的工作</li>
</ul>
<p>因此，软件开发项目需要自主型团队，让知识工作者实现自我管理。</p>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第三讲.pdf</code>——自主团队。</p>
</blockquote>

</details>
