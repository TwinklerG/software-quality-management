## 简答题

### 1. 为何说将"规范方法"、"计划驱动方法"等特征作为敏捷方法的对立面带有很大的误导性质？如何通过多种维度改进这种对各类开发过程的理解？（10分）


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>为何带有误导性质</strong>：</p>
<p>敏捷的核心是敏捷目的、敏捷价值观和敏捷原则。这些与"规范"和"计划驱动"并不矛盾。</p>
<p>影响敏捷与规范方法选择的维度：</p>
<ol>
<li>如果只有强有力的规范而缺乏敏捷，将导致官僚作风，进而停滞不前。团队将陷入为了测量而测量的处境之中，缺乏创新。</li>
<li>缺乏规范的敏捷则如同一个新创公司在盈利之前的不负责任的狂热。</li>
<li><strong>计划驱动的开发人员必须敏捷，敏捷开发人员必须规范。</strong></li>
</ol>
<p>敏捷和规范方法是互补的，而非对立的。过分强调对立会阻碍团队采用适合自身情况的最佳实践组合。</p>
<p><strong>改进理解的方式</strong>：</p>
<ul>
<li>从敏捷目的、价值观和原则出发理解敏捷</li>
<li>认识五种影响方法选择的维度（人员、文化、规模、关键性、变更率）</li>
<li>基于风险分析平衡敏捷与规范</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第二讲.pdf</code>和<code>assets/slides/8敏捷概述.pdf</code>。</p>
</blockquote>

</details>

### 2. 自主团队有哪些特点？为什么说这样的团队可以满足软件开发对团队的要求？


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>自主团队的特点</strong>：</p>
<ol>
<li>自行定义项目的目标</li>
<li>自行决定团队组成形式以及成员的角色</li>
<li>自行决定项目的开发策略</li>
<li>自行定义项目的开发过程</li>
<li>自行制定项目的开发计划</li>
<li>自行度量、管理和控制项目工作</li>
</ol>
<p><strong>为何满足软件开发要求</strong>：
软件开发是一种智力活动，开发者是智力劳动者。对于智力劳动者而言，管理的第一准则就是智力劳动者不能被管理，只能实现自我管理。软件开发需要：</p>
<ul>
<li>处理和讨论非常抽象的概念</li>
<li>把不同的部分（不可见）整合成一个可以工作的正确的系统</li>
<li>全身心地参与</li>
<li>努力做出卓越的工作</li>
</ul>
<p>自主团队赋予知识工作者必要的自主权，使其能够自我管理，从而充分发挥创造力和专业能力。</p>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第三讲.pdf</code>——自主团队与知识工作管理。</p>
</blockquote>

</details>

### 3. 马斯洛的"人的需求层次理论"描述的需求层次有哪几个？这样分层对软件开发有什么启发？（6分）


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>马斯洛需求层次（从低到高）</strong>：</p>
<ol>
<li>生理需求（Physiological）</li>
<li>安全需求（Safety）</li>
<li>社交需求（Love/Belonging）</li>
<li>尊重需求（Esteem）</li>
<li>自我实现需求（Self-actualization）</li>
</ol>
<p><strong>对软件开发的启发</strong>：</p>
<ul>
<li>软件开发是智力工作，开发者通常追求较高层次的需求（尊重和自我实现）</li>
<li>激励手段应该对应团队成员当前所处的需求层次</li>
<li>低层次需求未满足时，高层次激励手段可能失效（如：薪资问题未解决时，给予更多自主权可能效果不佳）</li>
<li>马斯洛晚年修正理论认为高层次需求不一定在低层次需求满足之前实现</li>
<li>对软件开发者而言，提供有挑战性的工作、认可其贡献、给予成长空间是有效的激励方式</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第三讲.pdf</code>——团队动力学中的激励理论。</p>
</blockquote>

</details>

### 4. 应对风险的典型策略有哪些？请举例说明


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p>典型风险应对策略：</p>
<ol>
<li><strong>规避（Avoidance）</strong>：改变项目计划以消除风险。例如，采用成熟技术替代不稳定的新技术。</li>
<li><strong>转移（Transfer）</strong>：将风险影响转移给第三方。例如，购买保险、外包高风险模块。</li>
<li><strong>缓解（Mitigation）</strong>：降低风险发生的概率或影响。例如，增加评审环节降低缺陷风险。</li>
<li><strong>接受（Acceptance）</strong>：有意识地承担风险后果。例如，对于低概率低影响的风险制定应急预案。</li>
</ol>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第四讲.pdf</code>——风险管理。</p>
</blockquote>

</details>

### 5. 为了确保最终软件产品的质量，在项目计划阶段应该注意哪些问题？（4分）


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<ol>
<li><strong>安排充分的质量活动</strong>：在计划中明确安排评审活动（个人评审和小组评审），确保评审时间充足</li>
<li><strong>设定质量指标目标</strong>：按照A/FR、PQI等质量指标要求制定计划</li>
<li><strong>重视设计阶段</strong>：设计是质量的基础，需要在计划中给予充分的设计时间</li>
<li><strong>参考历史数据</strong>：利用历史质量数据进行质量规划</li>
</ol>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第五讲.pdf</code>——质量计划。</p>

### 7. 请举例说明验证和确认的区别和联系（4分）


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>区别</strong>：</p>
<ul>
<li><strong>验证（Verification）</strong>：确保产品被正确建造——符合产品需求和产品组件需求。例如：代码评审检查代码是否符合设计规格。</li>
<li><strong>确认（Validation）</strong>：确保建造了正确的产品——满足客户需求。例如：验收测试检查产品是否真正解决了客户的问题。</li>
</ul>
<p><strong>联系</strong>：</p>
<ul>
<li>两者相互依存，都是为了提升最终产品质量</li>
<li>验证活动为确认活动提供前提条件——如果产品没有按规格建造，讨论它是否满足客户需求没有意义</li>
<li>验证依据来源于确认目标——产品组件需求必须与客户需求一致</li>
</ul>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第五讲.pdf</code>——V&amp;V。</p>
</blockquote>

### 8. 请解释规模估算和资源估算中估算偏差含义之间的差异，并据此简要列举对软件开发活动的启发


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>差异</strong>：</p>
<ul>
<li><strong>规模估算偏差</strong>：产生原因相对客观（如需求理解偏差、分解粒度不同等），偏差可以用来修正新的估算结果，具有较好的可参考性</li>
<li><strong>资源估算偏差</strong>（特别是时间估算）：产生原因更复杂，一方面与规模有关，另一方面与人的主观能动性有关。时间估算偏差可能是"自我实现的预言"——估算多少时间就按多少时间完成</li>
</ul>
<p><strong>启发</strong>：</p>
<ol>
<li>规模估算可以更多依赖历史数据和统计方法</li>
<li>资源/时间估算需要更谨慎，注意生产效率的个体差异</li>
<li>应区分对待规模估算和时间估算的准确性要求</li>
<li>建立过程数据收集机制，为未来估算积累高质量历史数据</li>
</ol>
<blockquote>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第四讲.pdf</code>——估算偏差分析。</p>
</blockquote>

</details>
