### 5. 对比TSP和SCRUM，下列说法不恰当的是

A. 都是过程框架，需要填补具体实践之后才是一个可以工作的过程
B. 一种是计划驱动方法，另外一种是敏捷方法
C. SCRUM适合迭代式场景，TSP适合瀑布场景
D. 两种方法都需要进行度量数据收集、分析，从而支持管理决策


<details>
<summary>点击查看答案与解析</summary>

<p><strong>答案：B、C</strong></p>
<blockquote>
<p><strong>题目解析</strong>：
- A 正确：TSP 和 Scrum 都是过程框架（Process Framework），需要填补具体工程实践（如编码规范、测试策略等）之后才是一个完整可工作的过程。

- C 不恰当：两者<strong>都适合迭代式场景</strong>。TSP 采用迭代式开发周期，并非仅适用于瀑布场景。
- D 正确：两者都强调度量的重要性，都需要收集和分析数据以支持管理决策。</p>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第三讲.pdf</code>——TSP 和 Scrum 对比。</p>

### 7. 以下关于马斯洛的需求层次理论描述不正确的是

A. 自我实现是寻求自尊（Esteem）
B. 激励来自为没有满足的需求而努力奋斗
C. 低层次的需求必须在高层次需求满足之前得到满足
D. 满足高层次的需求的途径比满足低层次的途径更少


<details>
<summary>点击查看答案与解析</summary>

<p><strong>答案：A、C、D</strong></p>
<blockquote>
<p><strong>题目解析</strong>：
- A 不正确：自我实现（Self-Actualization）是马斯洛需求层次的<strong>第五层（最高层）</strong>，而自尊（Esteem）是<strong>第四层</strong>，两者不可混为一谈。
- B 正确：激励的基本原理就是来自为尚未满足的需求而努力奋斗。
- C 不正确："必须"一词过于绝对。马斯洛晚年修正了自己的理论，认为高层次需求不一定在低层次需求<strong>完全</strong>满足之后才出现。
- D 不正确：满足高层次需求的途径比满足低层次的途径<strong>更为广泛</strong>（更多而非更少）。</p>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第三讲.pdf</code>——马斯洛需求层次理论。</p>

### 8. 以下关于团队动力学的论述，不恰当的是

A. 马斯洛的需求层次理论可以用来更好地维持激励水平
B. 智力工作的激励方式中，应该尽可能使用鼓励承诺这种方式
C. 麦克勒格的X理论适合用马斯洛底层需求激励
D. 海兹伯格的激励理论区分为内在因素和外在因素两种


<details>
<summary>点击查看答案与解析</summary>

<p><strong>答案：A</strong></p>
<blockquote>
<p><strong>题目解析</strong>：
- A 不恰当：马斯洛的需求层次理论可用于<strong>指导激励手段的选择</strong>（如针对不同需求层次设计不同激励方式），而非"维持激励水平"。维持激励水平更适合用 Herzberg 双因素理论来指导。
- B 正确：智力工作中鼓励承诺（Commitment）是有效的激励方式。
- C 正确：X 理论的员工关注底层需求（生理、安全），适合用马斯洛底层需求（薪酬、工作保障）来激励。
- D 正确：Herzberg 将激励因素分为内在因素（成就感、认可、成长）和外在因素（薪酬、工作环境、公司政策）。</p>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第三讲.pdf</code>——团队动力学（马斯洛、McGregor X-Y、Herzberg 双因素）。</p>

### 10. 下述关于WBS的描述中，哪些说法不正确的？

A. WBS应该对应OBS
B. WBS提供了范围管理的基础
C. WBS工作分解最底层的要素是实现目标的充分必要条件
D. WBS分解的时候，同一层不能应用不同标准


<details>
<summary>点击查看答案与解析</summary>

<p><strong>答案：A、D</strong></p>
<blockquote>
<p><strong>题目解析</strong>：
- A 不正确：WBS（工作分解结构）和 OBS（组织分解结构）关注角度和目的不同，WBS 不一定必须对应 OBS。

- D 不正确：WBS 分解时同一层<strong>可以</strong>应用不同标准。虽推荐同层保持一致的分解标准，但并非绝对禁止使用不同标准。</p>

### 12. 关于PSP质量管理策略，下列说法中正确的是

A. 用缺陷管理替代质量管理，既有必要性，也有合理性
B. 基本无缺陷的开发是通过开展高质量的评审来实现的
C. 经过训练，评审是所有消除缺陷的手段当中最高效的
D. PSP质量策略主要解决的是外部质量，而非内部质量


<details>
<summary>点击查看答案与解析</summary>

<p><strong>答案：A、B、D</strong></p>
<blockquote>
<p><strong>题目解析</strong>：
- A 正确：用缺陷管理替代质量管理是 PSP 的核心策略。"质量"概念抽象难以直接度量，而缺陷是具体可计数的，可以作为质量的代理指标，既有必要性也有合理性。
- B 正确：PSP 追求基本无缺陷的开发，通过开展高质量的评审（而非依赖测试）来实现。
- C 错误：经过训练后，<strong>编译</strong>（而非评审）是所有消除缺陷手段中最高效的，因为编译可以发现语法和类型错误，反馈速度最快。
- D 正确：PSP 使用面向用户的视图，主要解决的是<strong>外部质量</strong>（用户能感知到的缺陷），通过控制缺陷来保障用户层面的质量体验。</p>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第五讲.pdf</code>——PSP 质量策略。</p>

### 13. 关于DRL，下列说法中不正确的是

### 3. 挣值管理的三种实现方式。分别是简单、中级以及高级。请分别描述上述三种方式的基本要点。（10分）


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<p><strong>简单实现</strong>：仅关注进度信息。建立WBS定义工作范围→为每项工作定义PV→按规则赋EV值。常用0-100原则和50-50原则。AC不对EV产生影响。</p>
<p><strong>中级实现</strong>：在简单实现基础上加入日程偏差计算。SV = EV − PV；SPI = EV / PV。</p>
<p><strong>高级实现</strong>：加入成本线（AC）和预测线（BAC），计算成本偏差和成本偏差指数。可以明确项目落后时间。</p>
<blockquote>
<p><strong>题目解析</strong>：三种实现的本质区别在于涉及的信息维度：
- <strong>简单实现</strong>：仅 EV（挣值），关注"完成了多少"。
- <strong>中级实现</strong>：EV + PV（计划价值），可计算进度偏差（SV/SPI），关注"进度如何"。
- <strong>高级实现</strong>：EV + PV + AC（实际成本）+ BAC（完成预算），可计算成本偏差（CV/CPI），关注"花了多少钱，还要多久"。</p>
<p>记忆技巧：简单→进度，中级→进度偏差，高级→进度+成本。</p>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第四讲.pdf</code>——EVM。</p>
</blockquote>

### 4. 软件项目规模估算基本要点有哪些？（10分）


<details>
<summary>点击查看答案与解析</summary>

<p><strong>解答</strong>：</p>
<ol>
<li><strong>尽可能划分详细一些</strong>：估算多个部件时，总误差比各部件误差总和小</li>
<li><strong>建立对结果的信心</strong>：估算本质上是猜测，追求一致性</li>
<li><strong>依赖数据</strong>：充分参考历史数据</li>
<li><strong>估算要的是过程，而非结果</strong>：估算过程是相关干系人达成一致共识的过程</li>
</ol>
<blockquote>
<p><strong>题目解析</strong>：本题考的是规模估算的基本要点。核心思想：
- 第 1 点：分而治之——越细越准（统计学原理：总误差 &lt; 各部分误差之和）。
- 第 2 点：估算的精度追求一致性（Consistency）而非准确性（Accuracy）。
- 第 3 点：估算不是凭空猜测，要基于历史数据（规模、生产率、缺陷率等）。
- 第 4 点：估算过程本身的价值——促使干系人沟通、统一理解、达成共识，结果反而是副产品。</p>
<p><strong>知识点出处</strong>：<code>assets/slides/软件质量与管理.第四讲.pdf</code>——规模估算。</p>
</blockquote>
